Occtet Curator – User Manual

1. Introduction

1.1 What is Occtet Curator?

Occtet Curator is a professional platform for analyzing, curating, and assessing SBOMs (Software Bill of Materials). An SBOM describes which software components a project consists of. This information forms the basis for: - Security assessments (CVEs, vulnerabilities) - License compliance - Copyright analysis - Supply chain risk assessment Occtet Curator breaks down an SBOM into individual, verifiable units and makes them manageable for human review. The central object in this process is the InventoryItem.

1.2 Core Concepts

Term Meaning
SBOM List of all software components of a project (e.g., CycloneDX or SPDX)
InventoryItem A logically grouped software component including files, licenses, copyrights, and CVEs
SoftwareComponent The actual software (e.g., OpenSSL 3.0.2)
Vulnerability / CVE A known security vulnerability
VEX A statement indicating whether a CVE is relevant for this project
AuditNote Documentation of your decision

2. Core Principle: How Occtet Works

Occtet Curator always works on a project basis: 1. Create a project 2. Import an SBOM 3. Analyze the project 4. Review InventoryItems 5. Validate files, licenses, and copyrights 6. Assess vulnerabilities (VEX) 7. Export results The system is designed to transform a raw SBOM into a verified, audit-ready software inventory.

3. Projects

3.1 Creating a Project

When creating a project, you specify: - Name - Working directory - Optional metadata such as version, customer, or product Into the project directory: - the source code - and its dependencies are copied or referenced. This directory is the basis for: - file analysis - copyright scanning - license checks

3.2 Importing an SBOM

You can import an SBOM (e.g., SPDX or CycloneDX). During import, you can configure whether AI workflows should be used for: - copyright detection - license attribution - file assignment After import, Occtet creates: - SoftwareComponents - InventoryItems - file assignments - initial vulnerability information

4. Dashboard

The dashboard is your security traffic light. It shows, for example: - number of critical CVEs - open vulnerabilities - percentage of reviewed files - risk trends This allows you to immediately see: "Is this project currently release-ready or a risk?"

5. SBOM Management

This is your main working area. You see: - on the left: the file tree - on the right: the InventoryItems Ideally, each file belongs to exactly one InventoryItem.

6. InventoryItems – The Core Element

An InventoryItem represents a software component. It contains: - SoftwareComponent (name, version, origin) - Associated files - Vulnerabilities (CVEs) - Licenses - Copyrights - AuditNotes - VEX entries Typical examples: - OpenSSL 3.0.2 - Log4j 2.17.1 - A proprietary project module

7. Working with Files

7.1 Code Viewer

You can open, read, and search any file and compare it with its InventoryItem. You can see: - license headers - copyright notices - indications of third-party code

7.2 Reviewed Status

A file is considered reviewed if: - it is assigned to the correct InventoryItem - its license has been checked - its copyrights have been checked - all findings have been documented After that, it is marked as reviewed.

8. AuditNotes

AuditNotes are your memory and your evidence. Here you document, for example: - why a license is acceptable - why a CVE is considered not relevant - why a copyright can be ignored They are essential for: - ISO audits - customer inquiries - legal evidence

9. Vulnerabilities & VEX

Occtet automatically detects which CVEs belong to a SoftwareComponent. You decide via VEX whether it is: - Affected - Not Affected - Under Investigation - Fixed These statements can be exported in a machine-readable format.

10. Views & Overviews

View Purpose
Inventory All InventoryItems
VEX Listing All VEX decisions
Licenses All licenses in the project

All lists can be filtered, sorted, and searched.

11. Export

You can export: - cleaned SBOMs - VEX documents - license reports - audit evidence This allows you to: - deliver to customers - integrate with CI/CD - serve authorities or auditors

12. Administration

Administrators can: - manage users - assign roles - configure AI workflows - automate import jobs - review logs
