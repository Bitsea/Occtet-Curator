Occtet Curator – Benutzerhandbuch

1. Einführung

1.1 Was ist Occtet Curator?

Occtet Curator ist eine professionelle Plattform zur Analyse, Kuration und Bewertung von SBOMs (Software Bill of Materials) und Erstellung von VEX Berichten. Eine SBOM beschreibt, aus welchen Softwarekomponenten ein Projekt besteht. Diese Informationen bilden die Grundlage für: - Sicherheitsbewertungen (CVEs, Schwachstellen) - Lizenz-Compliance - Copyright-Analysen - Risikoabschätzung in der Software-Lieferkette Occtet Curator zerlegt eine SBOM in einzelne, überprüfbare Einheiten und macht diese für Menschen prüfbar. Das zentrale Objekt dabei ist das InventoryItem.

1.2 Zentrale Konzepte

Begriff Bedeutung
SBOM Liste aller Softwarekomponenten eines Projekts (z.B. SPDX).
InventoryItem Eine Abbildung der Softwarekomponente inklusive Dateien, Lizenzen, Copyrights und CVEs.
SoftwareComponent Die eigentliche Software Komponente (z.B. OpenSSL 3.0.2).
Vulnerability / CVE Sicherheitsformat, das bestätigt, ob ein Softwareprodukt von einer Schwachstelle (CVE) betroffen ist oder nicht.
VEX Aussage, ob eine CVE für dieses Projekt relevant ist.
AuditNote Dokumentation deiner Entscheidungen.

2. Grundprinzip: Wie arbeitet man mit Occtet?

Occtet Curator arbeitet projektbasiert. Eine Organisation kann mehrere Projekte und User enthalten. : Der Admin legt Organisationen an und weist User der Organisation zu. Auch legt er in der Configuration den Pfad an, in dem der Source Code gespeichert wird. Der Curator geht nun wie folgt vor: 1. Projekt anlegen (Projekte) 2. SBOM importieren (Import -> SPDX Import oder wenn ORT Instanz verfügbar über den ORT Starter) 3. Im SBOMmgmt wird nun das Projekt ausgewählt und mit Hilfe der InventoryItems auf der rechten Seite und dem Filetree auf der linken Seite kuratiert. 4. Die Overview zeigt Schwachstellen, Lizenzen und Copyrights des Projektes 5. Die InventoryItems enthalten Daten zur SoftwareKomponente, nicht nur Name und Version, auch Schwachstellen, Copyrights und Lizenzen. Zusätzlich enthalten sie Files, zugehörig zur SoftwareKomponente oder alleinstehend mit geteilten Merkmalen (z.B. die gleiche Lizenz), und Audit Notes in denen Nachweise zu den Daten festgehalten werden. 6. Hier im Kontext zur SoftwareKomponente des InventoryItems können zu deren Schwachstellen VEX Berichte erstellt und heruntergeladen werden 7. Die kuratierte SBOM kann auf der Projektseite zum jeweiligen Projekt heruntergeladen werden Das System ist darauf ausgelegt, aus einer rohen SBOM ein geprüftes, auditfähiges Softwareinventar zu erzeugen. Im Folgenden werden die einzelnen Seiten detaillierter beschrieben.

3. Configuration (Admin)

3.1 Systempfad anlegen

Der Systempfad bestimmt, wo der Source Code zu den Projekten gespeichert wird. Dies wird benötigt, um später im SBOMmgmt die Inhalte der Files anzuzeigen. Ohne Systempfad kann kein Projekt angelegt werden.

3.2 Textvervollständigung (Vorschläge)

By Default sind einige Vorschläge zur Textvervollständigung angelegt. Diese können in den AuditNotes oder für den InventoryItem Name benutzt werden. Nur der Admin kann neue Anlegen, zurzeit existieren nur zwei Kontexte, die zwingend so übernommen werden sollten und nur für diese zwei Anwendungen implementiert sind.

3.3 Suchbegriffe

Hier können Profile für Suchbegriffe erstellt werden, die dann in den File Inhalten hervorgehoben werden. Beliebte Suchbegriffe sind beispielsweise: itext, gpl, lgpl, stackoverflow. Die Suchbegriffe werden nacheinander mit Komma getrennt angegeben.

4. Organisation (Admin)

Der Admin erstellt Organisationen und weist diesen User zu. Auch Projekte kann der Admin den Organisationen zuordnen. Wenn der Admin ein Projekt erstellt, kann dieser auswählen welche Organisation dem Projekt zugewiesen ist. Wenn ein Curator ein Projekt erstellt, ist die Organisation des Projekts automatisch die des Users.

5. Projekte

5.1 Projekt anlegen

Beim Anlegen eines Projekts sind diese Angaben zwingend: - Name - Version - Produktverantwortlicher Zusätzlich können noch eine Kontaktemail und ein Suchprofil hinzugefügt werden. Wenn der Admin ein Projekt erstellt, kann dieser auswählen welche Organisation dem Projekt zugewiesen ist. Wenn ein Curator ein Projekt erstellt, ist die Organisation des Projekts automatisch die des Users. Die ganzen Daten, die den Projekten einer Organisation angehören, können innerhalb einer Organisation wiederverwendet werden. Lizenzen und Copyrights sind von Organisationen unabhängig und können global über alle Organisationen genutzt werden.

6. SBOM importieren

Du kannst eine SBOM (momentan SPDX) mit dem FileUpload importieren. Beim Import kannst du auch konfigurieren, ob KI-Workflows für Folgendes genutzt werden: - Copyright-Erkennung - Lizenzzuordnung Nach dem Import erzeugt Occtet: - SoftwareComponents - InventoryItems - Vulnerabilities (CVEs) - Files - Lizenzen - Copyrights

7. ORT Run starten

Auf der Import Seite findest du auch einen Button für den Start eines ORT Runs. Hierbei ist es wichtig eine laufende ORT (Docker) Instanz im System eingebunden zu haben. In den Application.properties oder in live in einer .env Datei gibst du die Verbindungsdaten an. Bei den Konfigurationen beim ORt Run wählst du welche Art Repository du scannen möchtest, den Link dazu und die Version.

8. Dashboard

Das Dashboard ist deine Sicherheitsampel. Es zeigt u. a.: - Anzahl kritischer CVEs zu den verschiedenen Kategorien (Critical, High, Medium, Low) - wie viele SoftwareKomponenten Schwachstellen haben So erkennst du sofort: "Ist dieses Projekt aktuell releasefähig oder ein Risiko?"

9. SBOM-Management

Dies ist dein Hauptarbeitsbereich. Du siehst: - links: den Dateibaum - rechts: die InventoryItems - mittig: Overview (Schwachstellen, Lizenzen, Copyrights) Idealerweise gehört jede Datei genau zu einem InventoryItem.

10. InventoryItems – Das Herzstück

Ein InventoryItem repräsentiert eine Softwarekomponente. Es enthält: - SoftwareComponent (Name, Version, HerkunftsURL) - zugehörige Dateien/Files - Schwachstellen (CVEs) (hier können VEX-Einträge erstellt werden) - Lizenzen - Copyrights - AuditNotes zur Dokumentation deiner Entscheidungen InventoryItems sind die Grundlage für deine Bewertungen. Sie helfen dir, die SBOM in handhabbare Einheiten zu zerlegen und diese Einheiten zu prüfen.

11. Arbeiten mit Dateien

11.1 Code-Viewer

Du kannst jede Datei: - öffnen - lesen - durchsuchen - einem InventoryItem zuordnen Dabei siehst du beispielsweise rechtliche Hinweise wie: - Lizenzheader - Copyright-Hinweise - Hinweise auf Drittcode Diese sollten in den AuditNotes dokumentiert werden, damit du später nachvollziehen kannst, warum du eine Datei einem InventoryItem zugeordnet hast oder warum du eine Lizenz oder ein Copyright akzeptabel findest. Auch werden Exportrestriktionen oder Ähnliches in den AuditNotes dokumentiert, die du im Dateiinhalt gefunden hast. Die Suchbegriffe von deinem Profil werden farblich hervorgehoben.

11.2 Reviewed-Status

Eine Datei gilt als geprüft, wenn: - sie dem richtigen InventoryItem zugeordnet ist - ihre Lizenz geprüft wurde - ihre Copyrights geprüft wurden - alle Erkenntnisse dokumentiert sind Danach wird sie als reviewed markiert.

12. AuditNotes

AuditNotes sind dein Gedächtnis und Nachweis. Hier dokumentierst du z. B.: - warum eine Lizenz akzeptabel ist - warum eine CVE als nicht relevant gilt - warum ein Copyright ignoriert werden darf Sie sind essenziell für: - ISO-Audits - Kundenanfragen / Transparenz - rechtliche Nachweise

13. Schwachstellen & VEX

Occtet erkennt automatisch, welche CVEs zu einer SoftwareComponent gehören. Du entscheidest per VEX, ob sie: - Affected - Not Affected - Under Investigation - Fixed Diese Aussagen lassen sich maschinenlesbar als VEX (Vulnerability Exploitability Exchange) Format exportieren. Diese kannst du in der InventoryItemView (Detailansicht im SBOMmgmt) für jede CVE einzeln oder zusammen erstellen.

14. Ansichten & Übersichten

Occtet bietet verschiedene globale Views:
Ansicht Zweck
Inventory Alle InventoryItems
VEX Listing Alle VEX-Entscheidungen
Licenses Alle Lizenzen im Projekt
SoftwareComponents Technische Komponenten
Projects Alle Projekte
Alle Listen lassen sich filtern, sortieren und durchsuchen.

15. Export

Du kannst exportieren: - bereinigte SBOMs (mit rechtlich relevanten Informationen, z. B. CVEs, Lizenzen, Copyrights) - VEX-Dokumente Damit kannst du: - Kunden beliefern - Behörden oder Auditoren bedienen

16. Integration

Durch die API mit ORT kann der Curator in eine Build-Pipeline integriert werden. So können automatisierte ORT Runs gestartet und die Ergebnisse in Occtet Curator importiert werden. Auch von ORt aus gestartete Runs werden vom Curator erkannt und können direkt importiert werden. So kannst du kontinuierlich deine SBOMs aktuell halten und Schwachstellen bewerten.

6. Importing an SBOM

You can import an SBOM (currently SPDX format) using the file upload feature. During the import process, you can also configure whether AI-powered workflows should be used for: - Copyright detection - License classification After the import, Occtet automatically generates: - SoftwareComponents - InventoryItems - Vulnerabilities (CVEs) - Files - Licenses - Copyrights

7. Starting an ORT Run

The Import page also provides a button to start an ORT run. To use this feature, a running ORT (Docker) instance must be connected to the system. Connection details can be configured either in the application.properties file or dynamically via a .env file. When configuring an ORT run, you define: - The type of repository to scan - The repository URL - The version to analyze

8. Dashboard

The dashboard acts as your project’s security indicator. It provides an overview of: - The number of CVEs categorized by severity (Critical, High, Medium, Low) - The number of software components affected by vulnerabilities This allows you to quickly assess whether a project is ready for release or poses a potential risk.

9. SBOM Management

This is your main workspace. The layout is structured as follows: - Left: File tree - Right: InventoryItems - Center: Overview (vulnerabilities, licenses, copyrights) Ideally, each file should be assigned to exactly one InventoryItem.

10. InventoryItems – The Core Element

An InventoryItem represents a software component and includes: - SoftwareComponent (name, version, source URL) - Associated files - Vulnerabilities (CVEs) (with the option to create VEX statements) - Licenses - Copyrights - AuditNotes to document decisions InventoryItems are the foundation of your analysis. They help break down the SBOM into manageable units that can be reviewed individually.

11. Working with Files

11.1 Code Viewer

For each file, you can: - Open it - Read it - Search within it - Assign it to an InventoryItem While reviewing files, you may encounter relevant legal information such as: - License headers - Copyright notices - References to third-party code These findings should be documented in AuditNotes to ensure traceability of your decisions. For example, you can record why a file was assigned to a specific InventoryItem or why a license or copyright is considered acceptable. Export restrictions or similar legal notes identified in the file content should also be documented in AuditNotes. Search terms defined in your profile are highlighted within the file content.

11.2 Reviewed Status

A file is considered reviewed when: - It is assigned to the correct InventoryItem - Its license has been verified - Its copyrights have been reviewed - All findings have been documented Once these steps are completed, the file can be marked as reviewed.

12. AuditNotes

AuditNotes serve as your documentation and audit trail. They are used to record, for example: - Why a license is considered acceptable - Why a CVE is classified as not relevant - Why a copyright notice can be ignored They are essential for: - Compliance audits (e.g., ISO) - Customer inquiries and transparency - Legal documentation

13. Vulnerabilities & VEX

Occtet automatically detects which CVEs are associated with each SoftwareComponent. You then assess their relevance using VEX statuses: - Affected - Not Affected - Under Investigation - Fixed These assessments can be exported in the machine-readable VEX (Vulnerability Exploitability Exchange) format. Within the InventoryItem view (detailed view in SBOM Management), VEX statements can be created either individually per CVE or in bulk.

14. Views & Overviews

Occtet provides several global views:
View Purpose
Inventory All InventoryItems
VEX Listing All VEX decisions
Licenses All licenses used in the project
SoftwareComponents Technical components
Projects All projects
All lists can be filtered, sorted, and searched.

15. Export

You can export: - Cleaned SBOMs (including legally relevant information such as CVEs, licenses, and copyrights) - VEX documents These exports can be used to: - Deliver results to customers - Provide documentation to authorities or auditors

16. Integration

Through its API integration with ORT, Occtet Curator can be embedded into a build pipeline. This enables: - Automated execution of ORT runs - Direct import of results into Occtet Curator ORT runs initiated externally are also detected by the Curator and can be imported seamlessly. This allows you to continuously keep your SBOMs up to date and systematically assess vulnerabilities.
