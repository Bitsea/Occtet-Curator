Occtet Curator – Benutzerhandbuch

1. Einführung

1.1 Was ist Occtet Curator?

Occtet Curator ist eine professionelle Plattform zur Analyse, Kuration und Bewertung von SBOMs (Software Bill of Materials) und Erstellung von VEX Berichten. Eine SBOM beschreibt, aus welchen Softwarekomponenten ein Projekt besteht. Diese Informationen bilden die Grundlage für: - Sicherheitsbewertungen (CVEs, Schwachstellen) - Lizenz-Compliance - Copyright-Analysen - Risikoabschätzung in der Software-Lieferkette Occtet Curator zerlegt eine SBOM in einzelne, überprüfbare Einheiten und macht diese für Menschen prüfbar. Das zentrale Objekt dabei ist das InventoryItem.

1.2 Zentrale Konzepte

Begriff Bedeutung
SBOM Liste aller Softwarekomponenten eines Projekts (z.B. SPDX).
InventoryItem Eine Abbildung der Softwarekomponente inklusive Dateien, Lizenzen, Copyrights und CVEs.
SoftwareComponent Die eigentliche Software Komponente (z.B. OpenSSL 3.0.2).
Vulnerability / CVE Sicherheitsformat, das bestätigt, ob ein Softwareprodukt von einer Schwachstelle (CVE) betroffen ist oder nicht.
VEX Aussage, ob eine CVE für dieses Projekt relevant ist.
AuditNote Dokumentation deiner Entscheidungen.

2. Grundprinzip: Wie arbeitet man mit Occtet?

Occtet Curator arbeitet projektbasiert. Eine Organisation kann mehrere Projekte und User enthalten. : Der Admin legt Organisationen an und weist User der Organisation zu. Auch legt er in der Configuration den Pfad an, in dem der Source Code gespeichert wird. Der Curator geht nun wie folgt vor: 1. Projekt anlegen (Projekte) 2. SBOM importieren (Import -> SPDX Import oder wenn ORT Instanz verfügbar über den ORT Starter) 3. Im SBOMmgmt wird nun das Projekt ausgewählt und mit Hilfe der InventoryItems auf der rechten Seite und dem Filetree auf der linken Seite kuratiert. 4. Die Overview zeigt Schwachstellen, Lizenzen und Copyrights des Projektes 5. Die InventoryItems enthalten Daten zur SoftwareKomponente, nicht nur Name und Version, auch Schwachstellen, Copyrights und Lizenzen. Zusätzlich enthalten sie Files, zugehörig zur SoftwareKomponente oder alleinstehend mit geteilten Merkmalen (z.B. die gleiche Lizenz), und Audit Notes in denen Nachweise zu den Daten festgehalten werden. 6. Hier im Kontext zur SoftwareKomponente des InventoryItems können zu deren Schwachstellen VEX Berichte erstellt und heruntergeladen werden 7. Die kuratierte SBOM kann auf der Projektseite zum jeweiligen Projekt heruntergeladen werden Das System ist darauf ausgelegt, aus einer rohen SBOM ein geprüftes, auditfähiges Softwareinventar zu erzeugen. Im Folgenden werden die einzelnen Seiten detaillierter beschrieben.

3. Configuration (Admin)

3.1 Systempfad anlegen

Der Systempfad bestimmt, wo der Source Code zu den Projekten gespeichert wird. Dies wird benötigt, um später im SBOMmgmt die Inhalte der Files anzuzeigen. Ohne Systempfad kann kein Projekt angelegt werden.

3.2 Textvervollständigung (Vorschläge)

By Default sind einige Vorschläge zur Textvervollständigung angelegt. Diese können in den AuditNotes oder für den InventoryItem Name benutzt werden. Nur der Admin kann neue Anlegen, zurzeit existieren nur zwei Kontexte, die zwingend so übernommen werden sollten und nur für diese zwei Anwendungen implementiert sind.

3.3 Suchbegriffe

Hier können Profile für Suchbegriffe erstellt werden, die dann in den File Inhalten hervorgehoben werden. Beliebte Suchbegriffe sind beispielsweise: itext, gpl, lgpl, stackoverflow. Die Suchbegriffe werden nacheinander mit Komma getrennt angegeben.

4. Organisation (Admin)

Der Admin erstellt Organisationen und weist diesen User zu. Auch Projekte kann der Admin den Organisationen zuordnen. Wenn der Admin ein Projekt erstellt, kann dieser auswählen welche Organisation dem Projekt zugewiesen ist. Wenn ein Curator ein Projekt erstellt, ist die Organisation des Projekts automatisch die des Users.

5. Projekte

5.1 Projekt anlegen

Beim Anlegen eines Projekts sind diese Angaben zwingend: - Name - Version - Produktverantwortlicher Zusätzlich können noch eine Kontaktemail und ein Suchprofil hinzugefügt werden. Wenn der Admin ein Projekt erstellt, kann dieser auswählen welche Organisation dem Projekt zugewiesen ist. Wenn ein Curator ein Projekt erstellt, ist die Organisation des Projekts automatisch die des Users. Die ganzen Daten, die den Projekten einer Organisation angehören, können innerhalb einer Organisation wiederverwendet werden. Lizenzen und Copyrights sind von Organisationen unabhängig und können global über alle Organisationen genutzt werden.

5.2 SBOM importieren

Du kannst eine SBOM (momentan SPDX) importieren. Beim Import kannst du konfigurieren, ob KI-Workflows für Folgendes genutzt werden: - Copyright-Erkennung - Lizenzzuordnung - Dateizuweisung Nach dem Import erzeugt Occtet: - SoftwareComponents - InventoryItems - Dateizuordnungen - erste Schwachstelleninformationen

6. Dashboard

Das Dashboard ist deine Sicherheitsampel. Es zeigt u. a.: - Anzahl kritischer CVEs - offene Schwachstellen - Anteil geprüfter Dateien - Risikotrends So erkennst du sofort: "Ist dieses Projekt aktuell releasefähig oder ein Risiko?"

5. SBOM-Management

Dies ist dein Hauptarbeitsbereich. Du siehst: - links: den Dateibaum - rechts: die InventoryItems Idealerweise gehört jede Datei genau zu einem InventoryItem.

6. InventoryItems – Das Herzstück

Ein InventoryItem repräsentiert eine Softwarekomponente. Es enthält: - SoftwareComponent (Name, Version, Herkunft) - zugehörige Dateien - Schwachstellen (CVEs) - Lizenzen - Copyrights - AuditNotes - VEX-Einträge Typische Beispiele: - OpenSSL 3.0.2 - Log4j 2.17.1 - eigenes Projektmodul

7. Arbeiten mit Dateien

7.1 Code-Viewer

Du kannst jede Datei: - öffnen - lesen - durchsuchen - mit dem InventoryItem vergleichen Dabei siehst du u. a.: - Lizenzheader - Copyright-Hinweise - Hinweise auf Drittcode

7.2 Reviewed-Status

Eine Datei gilt als geprüft, wenn: - sie dem richtigen InventoryItem zugeordnet ist - ihre Lizenz geprüft wurde - ihre Copyrights geprüft wurden - alle Erkenntnisse dokumentiert sind Danach wird sie als reviewed markiert.

8. AuditNotes

AuditNotes sind dein Gedächtnis und Nachweis. Hier dokumentierst du z. B.: - warum eine Lizenz akzeptabel ist - warum eine CVE als nicht relevant gilt - warum ein Copyright ignoriert werden darf Sie sind essenziell für: - ISO-Audits - Kundenanfragen - rechtliche Nachweise

9. Schwachstellen & VEX

Occtet erkennt automatisch, welche CVEs zu einer SoftwareComponent gehören. Du entscheidest per VEX, ob sie: - Affected - Not Affected - Under Investigation - Fixed Diese Aussagen lassen sich maschinenlesbar exportieren.

10. Ansichten & Übersichten

Occtet bietet verschiedene globale Views:
Ansicht Zweck
Inventory Alle InventoryItems
VEX Listing Alle VEX-Entscheidungen
Licenses Alle Lizenzen im Projekt
SoftwareComponents Technische Komponenten
Projects Alle Projekte
Alle Listen lassen sich filtern, sortieren und durchsuchen.

11. Export

Du kannst exportieren: - bereinigte SBOMs - VEX-Dokumente - Lizenzreports - Auditnachweise Damit kannst du: - Kunden beliefern - CI/CD integrieren - Behörden oder Auditoren bedienen

12. Administration

Administratoren können: - Benutzer verwalten - Rollen vergeben - KI-Workflows konfigurieren - Importjobs automatisieren - Logs prüfen
